# LiteCut：一个“不追求帧精确”的 Android 长视频快剪实验

今天这次 OpenSpec 实战很有意思，因为它不是为了做一个更复杂的视频编辑器，而是反过来问了一个更朴素的问题：

> 对长视频来说，用户真的那么在乎导出点位精确到某一帧吗？

很多时候答案是否定的。

一段一小时、两小时的视频，剪出来早一秒、晚两秒，通常并不影响使用。真正影响体验的是：导出太慢。只要一开始走上重编码路线，手机上很容易从“等一下”变成“等到不想用了”。

LiteCut 的 idea 就是抓住这个取舍：牺牲一点点边界精度，换来极快的无损导出。

## 这个 idea 为什么成立

传统视频编辑器追求的是精确：帧级时间线、复杂效果、多轨道、转场、字幕、调色、重编码。

LiteCut 反过来选择一个很窄的场景：

- 长视频；
- 基础切割、复制、删除、合并；
- 用户主要想快点把结果导出来；
- 边界早一两秒、晚一两秒可以接受；
- 不做滤镜、不做特效、不做复杂合成。

在这个场景里，最高价值不是“更专业”，而是“更快完成”。

所以 LiteCut 的核心不是把手机变成剪映，也不是把 FFmpeg 包装成一个全能编辑器，而是把编辑操作转换成可验证的 FFmpeg stream-copy 计划。

也就是尽量使用：

```text
-c copy
```

不重编码，不缩放，不加滤镜。

真正的剪切边界交给关键帧对齐策略处理。这样导出不是重新生产一条视频，而更像是把已有视频流按合理边界重新拼接。

这就是速度来源。

## Android 上最现实的问题

在桌面端，我们可以直接调用 `ffmpeg` 命令。

Android 不一样。App 不能简单假设系统里有一个可调用的 `ffmpeg` 可执行文件，也不能随便访问文件路径。

所以 LiteCut 做了几件很务实的事：

1. 通过系统文件选择器导入视频。
2. 能持久化读取权限就持久化。
3. 立即用用户选择的 URI 做预览，不等后台准备。
4. 后台把视频复制到 app 私有 workspace。
5. 在本地路径上跑 FFmpegKit / ffprobe 相关逻辑。
6. 导出完成后再发布结果。

这个流程很 Android，也很现实。

它没有幻想“像 Windows 一样随便调命令”，而是把 Android 的权限模型、URI 模型和 native media 工具链缝在一起。

## OpenSpec 带来的价值

这次实战最有意义的地方，是我们没有一上来就乱写。

我们先把问题拆成 change：

- `litecut-mvp`：验证 Android 文件访问、FFmpegKit、基础无损操作；
- `android-editor-timeline-ui`：把工具变成一个真正可操作的中文时间线编辑器；
- `android-preview-timeline-polish`：解决长视频预览黑屏、时间轴缩略图、缩放、滚动和 source-time 映射。

OpenSpec 的好处不是“文档变多”，而是每个阶段都能把目标、非目标、任务和验证方式说清楚。

比如我们明确写下非目标：

- 不做帧精确编辑；
- 不做重编码；
- 不做滤镜；
- 不做多轨；
- 不做密集逐帧缩略图。

这让项目没有被“视频编辑器应该什么都有”的惯性拖走。

LiteCut 的价值恰恰来自克制。

## 最关键的一次调试：有声音，没画面

后面做时间线 UI 时，我们遇到一个很典型的 Android 多媒体问题：

一小时视频导入后，声音马上出来了，但预览区域是黑的。

直觉上很容易怀疑 codec、视频文件、解码失败。

但 logcat 显示并不是这样：

- Android 的硬解码器在工作；
- render fps 正常；
- SurfaceView 也在 queue buffer；
- 本地抽帧验证视频本身不是黑屏。

也就是说，视频不是没解码，而是 `VideoView` / `SurfaceView` 在这个编辑器布局里的显示路径不可靠。

最后我们把预览层换成 Media3 / ExoPlayer，并使用 TextureView-backed rendering path。问题解决：长视频仍然可以马上听到声音，现在预览也能看到画面。

这一步很关键，因为它把 LiteCut 从“命令行工具套壳”推进到了“手机上真的能用的编辑器”。

## 时间线也必须服务于“快”

用户不需要一个专业 NLE，但需要知道自己在剪哪里。

所以我们加了：

- 单轨时间线；
- 片段选择；
- 切割、复制、删除、前移、后移；
- 播放进度条；
- 时间轴缩放；
- 横向滚动条；
- 稀疏缩略图。

缩略图也遵守同一个原则：够用就好，不做逐帧。

短片段一两张，普通片段几张，超长片段最多有限数量。生成失败也不影响编辑，时间线退回中性片段块。

这套 UI 看起来像一个经典视频编辑器，但内部精度和复杂度都被刻意弱化。它不是为了炫技，而是为了让用户更快找到片段、更快导出。

## 最核心的工程取舍

LiteCut 的工程取舍可以概括成一句话：

> 不用重编码解决的问题，就不要重编码。

这带来几个直接收益：

- 导出速度快；
- 手机发热少；
- 失败面更小；
- 长视频体验更好；
- MVP 能很快跑通。

当然，它也有代价：

- 边界不追求帧级精确；
- 某些格式组合可能不能直接 merge；
- 复杂效果不在范围内；
- 用户需要接受“快剪”而不是“精剪”。

但这不是缺点，这是产品定位。

## 开源项目

项目已整理为独立 Android 仓库：

```bash
gh repo clone STEPHENXING/LiteCut-android
```

仓库里只放 LiteCut Android 项目本身：

- Android app 源码；
- core timeline / export planning 逻辑；
- FFmpegKit adapter；
- Media3 preview adapter；
- 单元测试和仪表化测试；
- 项目 README 和说明文章。

OpenSpec 的 change、proposal、tasks 等工作流材料不混入项目仓库。它们属于开发过程，不属于 LiteCut Android 代码本体。

## 为什么我喜欢这个项目

因为它不是“再做一个剪映”。

它是一个非常明确的小工具：

给长视频做基础剪辑，允许边界粗一点，但导出必须快。

这背后有一个很现实的判断：很多用户真正需要的不是帧级精确，而是少等几分钟、几十分钟。

LiteCut 现在还只是 MVP，但这个 MVP 的方向是成立的。

它抓住了一个被专业视频编辑器容易忽略的场景：长视频快剪。

这也是今天这次 OpenSpec 实战最让我满意的地方。我们不是为了写规范而写规范，而是用规范把一个有锋利取舍的 idea，一步步做成了能在 Android 真机上跑起来的东西。
